# 요구사항 명세서: 카메라 관제 클라이언트 (인터페이스 프로토콜 포함)

## 1. 개요
- **목적**: 서버-클라이언트 구조에서 카메라 관제(제어 및 스트리밍 재생)를 수행하는 클라이언트 애플리케이션의 요구사항을 정리한다. 본 문서는 스트리밍 인터페이스(RTSP/H.264)와 제어용 인터페이스(명세화된 제어 프로토콜)를 함께 정의하여, 구현과 테스트에 필요한 기술 사양을 제공한다.
- **대상 시스템**: 카메라 관제 시스템 (Client-Server 구조)

## 2. 용어 정의
- **RTSP/H.264**: 서버-클라이언트 비디오 스트리밍 표준(RTSP 세션 관리, RTP/H.264 페이로드)
- **Control Protocol**: 본 문서에서 정의하는 제어 명령/응답 포맷(예: JSON-over-TCP 또는 WebSocket)을 의미

## 3. 기능 요구사항
- **연결 제어**: 클라이언트는 서버에 대해 `접속` 및 `접속 해제` 제어 명령을 전송할 수 있어야 한다(제어 프로토콜 사용).
- **화면(재생) 제어**: 사용자는 재생 중인 영상의 `frame_rate` 및 `resolution`을 변경하는 명령을 전송할 수 있어야 하며, 서버로부터 변경 결과(성공/실패)를 수신해야 한다.
- **스트리밍 재생**: 서버로부터 송출되는 RTSP/H.264 스트림을 수신하여 실시간으로 화면에 재생할 수 있어야 한다(RTP 패킷 손실 복구/재시도 로직 권장).
- **명령 송출 및 응답 처리**: 모든 제어 명령은 Control Protocol을 통해 송출되며, 명확한 JSON 응답(상태 코드 및 메시지)을 통해 성공 여부를 확인해야 한다.
- **에러 처리 및 알림**: 문제 발생 시 화면에 명확한 에러 메시지를 표시하고, 상세 로그를 남기며(로그 레벨: INFO/WARN/ERROR), 오류 분류(code)와 권장 조치(예: 재시도, 네트워크 점검)를 사용자에게 안내한다.

## 4. 인터페이스 프로토콜 (명세)
본 절은 `interface-protocol` 문서가 없을 경우를 대비한 표준 사양로, 구현 시 서버와 합의된 세부사항으로 대체 가능하다.

4.1. 스트리밍 인터페이스 (RTSP/H.264)
- 지원 형식: RTSP 1.0, RTP/H.264 페이로드
- 예시 스트림 URL: `rtsp://<host>:554/stream/<camera-id>`
- 전송 방식: RTP over UDP(기본) 또는 RTP over TCP(터널링/인터리브드)
4.2. 제어 인터페이스 (Control Protocol) — JSON over TCP (기본 권장)
- 전송 계층: TCP (안정성 보장), 포트 예시: 9000 (환경에 따라 변경 가능)
- 메세지 포맷: UTF-8 JSON 객체
요청(Request) 예시:
```json
{
  "type": "command",
  "command": "connect",
  "params": { "rtsp_url": "rtsp://10.0.0.2:554/stream/1" },
  "request_id": "r-12345"
}
```

응답(Response) 예시:
```json
{
  "type": "response",
  "request_id": "r-12345",
  "status": 200,
  "message": "connected",
  "data": { "session_id": "s-98765" }
}
```

지원 명령 목록(예시):
- `connect`: RTSP 세션을 시작함. 파라미터: `rtsp_url`.
- `disconnect`: RTSP 세션 종료. 파라미터: `session_id` 또는 없음.
- `set_param`: 재생 파라미터 변경. 파라미터: `{ "frame_rate": 30, "resolution": "1280x720" }`.
- `get_status`: 현재 세션/스트림 상태 요청.
- `ping` / `health_check`: 연결 확인용.

상태 코드(간단 스킴):
- `200` — OK (성공)
- `400` — Bad Request (잘못된 파라미터)
- `401` — Unauthorized (인증 필요/실패)
- `404` — Not Found (리소스 없음)
- `500` — Internal Error (서버측 오류)

보안 및 인증:
- 기본 인증 방식: Token 기반(예: Bearer token) 또는 TLS로 보호된 WebSocket/HTTPS 사용 권장.
- 인증 토큰은 Control Protocol 메시지의 헤더 또는 JSON 필드(`auth_token`)를 통해 전달될 수 있다.

로그링 및 모니터링 인터페이스:
- 클라이언트는 `log` 이벤트를 서버로 전송할 수 있으며, 서버는 모니터링 엔드포인트로 메트릭(push) 전송을 지원하면 좋다.

4.3. 예외 및 재시도 정책
- 네트워크 오류(타임아웃/연결 끊김): 자동 재시도 백오프 정책(예: 1s, 2s, 5s) 적용
- 스트리밍 디코딩 오류: 재초기화 또는 사용자에게 재시도 안내

## 5. 비기능 요구사항
- **성능(지연/프레임)**: 실시간 관제가 가능하도록 낮은 지연(목표: 엔드투엔드 지연 200ms 이하, 환경에 따라 조정)과 안정적 프레임 재생을 보장해야 하며, 사용자가 설정한 프레임률을 준수해야 한다.
- **호환성**: 스트리밍은 RTSP/H.264 표준을 통해 수신 가능해야 한다. 제어는 위 명세된 JSON-over-TCP Control Protocol을 기본으로 하되, 서버 구현에 맞춰 WebSocket/HTTP API 연동을 허용한다.
- **신뢰성**: 네트워크 오류 발생 시 명확한 상태(Disconnected/Retrying/Failed)를 표시하고, 재시도 정책을 적용해야 한다.
- **확장성**: 향후 다른 코덱(예: H.265) 또는 프로토콜(예: WebRTC, gRPC) 추가를 고려한 모듈 구조여야 한다.

검증 항목(수락 기준):

- RTSP 스트림을 수신하여 UI에서 실시간 재생이 가능하다(플레이어가 H.264 NAL decoding 수행).
